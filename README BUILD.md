# KRAKEN — клиентский лендинг: инструкция по внедрению

Ты — Claude Code на VPS владельца детейлинг-студии KRAKEN (Краков).
Владелец не программист. Этот репозиторий — полное ТЗ новой фичи.

Читать в таком порядке:
1. Этот файл (как строить и куда встраивать).
2. `SPEC.md` (что строить — вся логика и все принятые решения).
3. `landing-template.html` (утверждённый дизайн страницы клиента).

## Существующая система (не изучать заново с нуля — вот карта)

- Каталог: `/home/ubuntu/kraken/`
- pm2-процессы: `kraken` (Telegram-бот, index.js + kraken_core.js),
  `retell-webhook` (AI-телефония, нативный http на порту 3001),
  `retell-tunnel-watchdog`. Плюс cron `watchdog.sh`.
- Данные: SQLite `kraken.db` — источник истины; Google Sheets — зеркало
  (mirror.js, сервис-аккаунт google.js). Фото — Google Drive, папка на
  машину (getOrderFolder), ссылки дублируются в лист «Фото».
- `kraken_core.js` (~4900 строк): SERVICES (коды услуг → name_ua/name_pl,
  stages[] с checklist[]), колонки листа «Авто» в `const A`
  (PLATE, MODEL, PHONE=4, TYPE=6 — коды услуг через запятую, STATUS=7,
  STAGE=8, WORKER=9...), статусы машин: «в очереди|в работе|готова к
  выдаче|выдана». Лист «Логи» — закрытые этапы со старт/финиш.
  Лист «Персонал»: PIN|Имя|telegram|роль|...
- `alertAdmin()` в index.js — уведомление админа в Telegram.

## Что нельзя сломать (просьба владельца — минимум правил, но эти три святые)

1. Перед любыми изменениями kraken.db и kraken_core.js — резервная копия
   (в каталоге уже есть практика *.bak).
2. Не менять calcPay() и любую логику денег/зарплат/касс.
3. Лендинг — ОТДЕЛЬНЫЙ новый pm2-процесс (порт 3002), не внутри бота.
   В kraken_core.js добавляются только маленькие хуки-вызовы.

## Секреты (.env, владелец добавит значения сам — попроси его в нужный момент)

- `SMSAPI_TOKEN` — OAuth-токен smsapi.pl (аккаунт создан; до верификации
  SMS уходят только на номер владельца — использовать для тестов).
  Имя отправителя «KRAKEN» — в процессе одобрения; до тех пор слать
  с дефолтного поля.
- `ANTHROPIC_API_KEY` — понадобится с фазы 4 (подписи фото) и 5 (чат).
  Владельца надо провести за руку по console.anthropic.com.
- `LANDING_BASE_URL=https://link.krakendetailing.pl`
- DNS: A-запись link.krakendetailing.pl → 57.128.226.103 уже создана
  в GoDaddy. Проверь `dig +short link.krakendetailing.pl`.

## SSL и вход из интернета

Рекомендация: Caddy как reverse proxy (сам получает и продлевает
Let's Encrypt): `link.krakendetailing.pl → localhost:3002`.
Проверь, открыты ли порты 80/443 (ufw / панель OVH); если закрыты —
объясни владельцу по шагам, где открыть.

## Фазы стройки (по одной, после каждой — показать владельцу результат)

Фаза 0. Разведка + бэкап. Прочитать index.js, kraken_core.js (точки:
создание заказа, смена этапа, загрузка фото, завершение машины).
Бэкап kraken.db.

Фаза 1. Веб-сервер лендинга (новый pm2 `kraken-landing`, порт 3002).
Таблица landing_tokens (token ↔ carId). Маршрут GET /t/:token →
страница по landing-template.html, но с реальными данными:
- этапы из SERVICES по кодам TYPE (кодов может быть несколько);
- названия этапов PL: сгенерировать словарь переводов UA→PL всех
  уникальных названий этапов из SERVICES + тексты «что происходит
  сейчас» (PL+UA) — файл config/stage_texts.json, показать владельцу;
- текущий этап из STAGE, время пройденных из «Логи»;
- фото до/после из листа «Фото»/Drive;
- мастер из WORKER + «Персонал» (фото мастеров: добавить колонку-ссылку,
  пока пусто — аватар-буква как в шаблоне);
- статусная логика видимости блоков как в шаблоне (data-stage).
Демо-панель из шаблона в проде убрать. Языки PL/UA — переключатель,
выбор запоминать (localStorage тут можно — это наш сайт, не артефакт).
Caddy + SSL. Тест: открыть ссылку реальной машины с телефона.

Фаза 2. SMS. Модуль sms.js (SMSAPI REST, token из .env). Хук в точке
создания заказа: если телефон валиден → SMS со ссылкой /t/:token.
Формат и текст SMS (PL) — короткий, с именем клиента если есть.
Тест на номер владельца.

Фаза 3. Push. Web-push (VAPID), таблица push_subscriptions, баннер
подписки уже в шаблоне. Хук на смену этапа → push. На статус
«готова к выдаче»/«выдана» → push + SMS-дубль (решение владельца:
2 SMS на клиента максимум).

Фаза 4. Подписи фото ИИ. Таблица photo_captions. Новое фото «до» →
Claude API (Haiku, vision): зона + короткая заметка PL+UA → мастеру
в Telegram предпросмотр с кнопками «Подтвердить / Править / Без
подписи» → подтверждённое видно клиенту. Плюс: перед загрузкой
«после» бот шлёт мастеру фото «до» этой машины (с подписями) — чтобы
снял те же места.

Фаза 5. Чат. Таблица chat_messages. Виджет уже в шаблоне. Сообщение
клиента → серверу → мастеру (WORKER) в Telegram + черновик ответа от
Claude (перевод PL/UA в обе стороны) с кнопками «Отправить / Править».
Ответ мастера → на страницу (поллинг или SSE). ИИ сам клиенту НЕ
отвечает — только черновики.

Фаза 6. Допуслуги + отзывы + коллаж.
- config/upsells.json (правило: не предлагать код, уже есть в TYPE;
  список пока из одного примера «антидождь» — владелец дополнит).
  Клик «Хочу»/«Нет» → таблица upsell_clicks + мгновенный alertAdmin
  (ID, номер, телефон, услуга). Цену/заказ НЕ менять.
- Отзывы: белая схема из SPEC.md (обе кнопки всем). Оценка → таблица
  reviews; 1–3 → мгновенный alertAdmin даже без текста.
- Коллаж: до/после + логотип + @krakendetailing (sharp), кнопка
  «Поделиться» через Web Share API с файлом.

Фаза 7. Имя клиента. В диалог приёма машины в боте добавить
необязательный вопрос «Имя клиента?» (кнопка «Пропустить»).
Хранить в новой колонке/таблице, использовать в приветствии и SMS.
